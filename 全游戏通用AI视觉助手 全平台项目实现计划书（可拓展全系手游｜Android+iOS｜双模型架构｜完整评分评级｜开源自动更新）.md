## 一、项目整体定位

项目名称：全游戏通用AI视觉助手（支持拓展全系手游，默认适配王者荣耀，可快速兼容其他手游）

核心模式：手机端实时屏幕流采集 \+ 设备端轻量AI识别 \+ 局域网PC多模态大模型深度分析

运行架构：手机只做采集与UI，PC（同路由器内网）承担所有AI大模型推理，同时兼容**全网云端AI供应商模型**，双模型架构自由切换

合规属性：纯画面视觉分析，不读取游戏数据、不修改游戏、不模拟操作、不触碰游戏内存，属于合规辅助工具。

## 二、完整技术栈明细（Android \+ 未来iOS）

### 1）手机端通用技术

- 屏幕流采集：MediaProjection（Android） / RPScreenRecorder（iOS 录屏流）

- 实时帧处理：YUV 原始像素流，无截图、无文件保存、无OCR轮询

- 端侧轻量AI：NCNN / MNN 部署 YOLOv8\-Nano、CRNN文字识别

- 全局UI：悬浮窗、全局置顶、游戏内图层渲染

- 语音系统：离线ASR语音识别 \+ 离线TTS语音播报

- 网络通信：WebSocket长连接（二进制帧）\+ 内网HTTP容错

- 视频编码：MediaCodec硬编码 H264/H265 增量帧推送

### 2）局域网PC服务端技术

- 模型服务：vLLM / Ollama / LM Studio（支持自定义IP端口）

- 多模态模型：Qwen\-VL / Llava / MiniCPM\-V

- 中转服务：FastAPI \+ WebSocket 帧接收与推理调度

- 自动配置：防火墙放行、内网IP自动绑定、端口监听、模型自动加载

### 3）核心算法技术

- 画面差分检测：只推送变化帧，静止画面0流量

- 事件驱动AI推理：只有团战、击杀、开龙、选人、对战状态变化才调用大模型

- 分层识别：端侧做90%简单识别，大模型只做10%深度分析

- 对局隔离机制：每局独立ID、自动清零、独立评分、独立报告

### 功能4：动态AI评分体系（全游戏通用 \+ 王者荣耀全套精准段位评级规则）

本项目采用**模块化评分引擎**，支持多游戏独立评分规则配置，可自由新增LOL手游、和平精英等游戏评分模板。项目默认深度适配**王者荣耀排位/巅峰赛**，内置官方对标、可直接代码落地的五位置、四段位完整评级判定逻辑。单局独立计分，新对局进入选人界面自动清零重置分数与评级数据，对局数据与评级全程绑定单局唯一ID。

通用评级分数区间（全局统一标准）：

- 0–59：无评级

- 60–74：铜牌

- 75–89：银牌

- 90–96：金牌

- 97–100：顶级

加分扣分采用**本地视觉识别基础行为加减分 \+ 大模型全局综合修正**双层判定逻辑，对局过程实时刷新分数，分数跨越段位阈值时，自动触发弹窗提示 \+ 语音播报双重提醒。结算阶段由多模态模型结合全局战局数据，最终校准本局总分与最终评级，保证评分贴合对局真实表现。

### 功能6：AI辅助\-屏幕实时共享\+DeepSeek\-OCR结构化画面分析

#### 6.1 功能定义与业务价值

**功能定义：**

基于 Android MediaProjection 系统级屏幕共享能力，实现游戏画面实时采集，通过 DeepSeek-OCR 视觉大模型对屏幕内容进行结构化识别与分析，将非结构化的游戏画面转换为可量化、可评分、可追溯的结构化数据，驱动实时对局评分、段位判定、战术建议等上层应用。

**核心能力：**

- **屏幕实时共享**：无文件IO、无截图保存、内存级像素流直传，延迟 < 200ms
- **DeepSeek-OCR 结构化识别**：精确识别游戏内所有文字元素（KDA、经济、等级、技能CD、装备、小地图信息等）
- **画面结构化解析**：将画面信息转换为 JSON 结构化数据，供评分引擎直接消费
- **双部署模式**：支持局域网本地模型（零费用、低延迟）与云端 API（开箱即用、高质量）

**业务价值：**

1. **全游戏通用适配**：基于 OCR 的通用识别方案，新增手游适配仅需配置关键词映射规则，无需重新训练模型
2. **高精度数据采集**：DeepSeek-OCR 对游戏UI文字识别准确率 > 98%，远超传统模板匹配方案
3. **纯视觉合规方案**：不读取游戏内存、不Hook、不注入，完全基于画面分析，绝对合规
4. **降本增效**：本地模型部署实现零Token消耗，云端API按需调用，双模式自由切换
5. **可扩展架构**：结构化输出可对接任意下游应用（评分引擎、战术分析、数据统计、复盘回放等）

#### 6.2 完整业务运行流程

```
用户开启AI辅助
      ↓
[1] 权限申请 → MediaProjection 录屏权限弹窗 → 用户授权
      ↓
[2] 实时画面采集 → ImageReader 持续获取屏幕帧 → 内存流转，无文件落地
      ↓
[3] 本地前置过滤
      ├─ 画面差分检测：静止画面跳过，变化帧才处理
      ├─ 区域裁剪：仅识别游戏UI核心区域，减少无效计算
      ├─ 帧率控制：对局中5秒1次，非对局中10秒1次，事件触发即时识别
      └─ 质量判断：模糊帧、黑屏帧自动丢弃
      ↓
[4] OCR模型推理
      ├─ 本地模式：WebSocket → 局域网PC → vLLM/Ollama → DeepSeek-OCR → 结构化JSON
      └─ 云端模式：HTTPS → 硅基流动/DeepSeek官方API → DeepSeek-OCR → 结构化JSON
      ↓
[5] 数据解析评分
      ├─ OCR结果解析：提取KDA、经济、等级、装备、时间等关键字段
      ├─ 数据校验：异常值过滤、连续性校验、前后帧一致性验证
      ├─ 评分引擎：根据游戏规则计算实时分数、段位评级
      └─ 事件检测：击杀、死亡、助攻、团战、控龙等关键事件识别
      ↓
[6] 悬浮UI展示
      ├─ 实时分数 + 段位徽章更新
      ├─ KDA / 经济 / 参团率等核心数据面板
      ├─ 关键事件弹幕式提示
      └─ 语音播报触发（击杀、段位变化、团战建议等）
```

**各环节详细说明：**

**环节1 — 权限申请**
- 首次使用弹出系统级 MediaProjection 授权弹窗
- 用户授权后自动启动前台服务，状态栏常驻通知
- 支持随时从通知栏停止录屏
- Android 10+ 自动申请前台服务类型 `mediaProjection`

**环节2 — 实时画面采集**
- 采用 `ImageReader.newInstance()` 创建图像读取器
- 格式：`PixelFormat.RGBA_8888`，分辨率可配置（默认 720p）
- 缓冲队列：2帧，保证最新帧优先
- 采集帧率：默认 10fps，可动态调整

**环节3 — 本地前置过滤（性能节流核心）**
- **画面差分**：与上一帧像素差异 < 2% 直接跳过
- **区域裁剪**：仅裁剪游戏UI核心区域（顶部状态栏、底部技能栏、左侧小地图等）
- **动态帧率**：
  - 大厅/匹配/结算：10秒/次
  - 对局中：5秒/次
  - 团战/击杀事件：即时识别（事件驱动）
- **质量过滤**：
  - 全黑/全白帧丢弃
  - 模糊度 > 阈值丢弃
  - 分辨率不达标丢弃

**环节4 — OCR模型推理**
- 输入：JPEG 压缩图片（质量 60-80% 可调）
- 输出：结构化 JSON，包含所有识别到的文字及坐标
- 平均响应时间：本地模型 < 500ms，云端API < 1.5s
- 支持并发控制：同时最多 1 个推理请求，请求排队

**环节5 — 数据解析评分**
- 字段提取器：根据游戏模板配置，从 OCR 结果中提取关键字段
- 数据清洗：数字格式化、单位转换、异常值剔除
- 一致性校验：与历史数据对比，突变值自动标记并二次验证
- 评分计算：调用评分引擎，实时更新分数与段位

**环节6 — 悬浮UI展示**
- 悬浮球：实时显示当前分数 + 段位
- 展开面板：KDA、经济、参团率、输出占比、评级详情
- 事件提示：击杀/死亡/助攻弹幕式弹窗
- 语音播报：关键事件 + 段位变化 + 战术建议

#### 6.3 双部署模式

##### 模式A：局域网本地 DeepSeek-OCR（推荐）

**架构：**
```
手机端（采集+UI） ←WebSocket→ 局域网PC（模型推理）
                         ↓
              vLLM / Ollama / LM Studio
                         ↓
              DeepSeek-OCR 本地模型
```

**部署要求：**
- PC配置：NVIDIA GPU（显存 ≥ 8GB 推荐） / 或 CPU（速度较慢）
- 网络：同一路由器局域网，延迟 < 5ms
- 模型：DeepSeek-OCR（约 7B 参数，量化版 4-bit 约 4GB）

**优势：**
- ✅ **零Token费用**：一次部署，永久免费使用
- ✅ **低延迟**：局域网传输 + 本地推理，响应 < 500ms
- ✅ **隐私安全**：画面数据不出内网，无数据泄露风险
- ✅ **高可用**：不依赖外网，断网也能正常使用
- ✅ **可定制**：可根据游戏优化提示词、调整识别策略

**劣势：**
- ❌ 需要 PC 设备 + 一定的技术配置门槛
- ❌ 对 GPU 显存有要求

##### 模式B：云端 DeepSeek OCR API

**架构：**
```
手机端（采集+UI） ←HTTPS→ 云端API服务商
                              ↓
                    DeepSeek-OCR 云端模型
```

**支持服务商：**
- **硅基流动（SiliconFlow）**：推荐，国内访问快，价格低
- **DeepSeek 官方**：官方API，稳定可靠
- 其他兼容 OpenAI 协议的服务商

**配置方式：**
- APP 内 → 模型页面 → 选择对应供应商 → 输入 API Key → 模型选择 "vision/视觉模型"
- 支持自定义 Base URL、模型名称、API Key

**优势：**
- ✅ **开箱即用**：无需部署，输入 Key 即可使用
- ✅ **高质量**：云端模型资源充足，识别效果稳定
- ✅ **无硬件要求**：手机即可，不需要 PC
- ✅ **自动更新**：模型自动保持最新版本

**劣势：**
- ❌ 产生 Token 费用（约 0.003-0.01 元/次识别）
- ❌ 依赖网络，延迟相对较高（1-3s）
- ❌ 数据需上传至云端

##### 模式切换

- APP 内一键切换，无需重启
- 自动检测：优先尝试本地，本地不可用自动降级云端
- 手动优先级：用户可设置"优先本地"或"仅云端"

#### 6.4 性能节流与容错机制

**性能节流策略：**

| 节流维度 | 策略 | 效果 |
|---------|------|------|
| **帧率控制** | 非对局 10s/次，对局中 5s/次，事件驱动即时识别 | 减少 70% 无效请求 |
| **画面差分** | 变化 < 2% 的帧直接跳过 | 静止画面 0 消耗 |
| **区域裁剪** | 仅裁剪核心UI区域，不做全图识别 | 减少 50% 推理计算量 |
| **图片压缩** | JPEG 质量 60-80%，分辨率 720p | 传输体积减少 80% |
| **请求合并** | 短时间内多次触发合并为一次 | 避免重复请求 |
| **并发控制** | 同时最多 1 个推理请求，队列长度 1 | 防止模型过载 |
| **缓存复用** | 相同画面结果缓存 3 秒 | 连续相似帧直接复用 |
| **动态降频** | CPU/温度过高时自动降低识别频率 | 防止手机过热 |

**容错机制：**

1. **网络异常**
   - 本地模式断连：自动重试 3 次（间隔 1s/2s/4s）
   - 重试失败：自动降级为纯端侧轻量识别
   - 网络恢复：自动恢复 OCR 识别

2. **模型推理失败**
   - 单帧失败：丢弃，等待下一帧，不影响整体流程
   - 连续 3 次失败：触发降级（降低分辨率/质量/频率）
   - 连续 10 次失败：暂停 OCR 识别 30 秒，语音提示

3. **数据异常**
   - 异常值检测：超出合理范围的数据自动标记
   - 前后帧校验：突变值需连续两帧确认才生效
   - 数据补全：短时缺失使用历史数据插值补全

4. **内存保护**
   - 帧队列最大长度：2帧，超出自动丢弃旧帧
   - Bitmap 自动回收：每帧处理完立即 recycle
   - 低内存预警：系统内存不足时自动降低识别频率

5. **服务保活**
   - 前台服务 + 常驻通知，防止系统杀死
   - START_STICKY 自动重启
   - 唤醒锁 + WiFi锁，保证后台稳定运行

#### 6.5 与现有模块的联动关系

**与王者荣耀段位判定引擎联动：**

```
屏幕OCR识别 → 提取KDA/经济/时间等数据
                ↓
        段位判定引擎（ScoringEngine）
                ↓
    ┌───────────┼───────────┐
    ↓           ↓           ↓
  实时分数    段位评级    事件检测
    ↓           ↓           ↓
  悬浮UI展示  语音播报    记录入库
```

- **数据输入**：OCR 识别结果是评分引擎的核心数据来源
- **事件驱动**：识别到击杀/死亡/助攻等事件时，触发评分引擎实时计算
- **校准机制**：每 30 秒 OCR 全量识别一次，校准端侧轻量识别的误差
- **结算复核**：对局结束时，OCR 全量识别结算界面，作为最终评分依据

**与悬浮球模块联动：**

- **实时数据更新**：OCR 识别到的 KDA、经济、评分等数据实时同步到悬浮球
- **状态指示**：悬浮球显示 OCR 运行状态（绿色=正常，黄色=降级，红色=异常）
- **快捷操作**：点击悬浮球可手动触发一次 OCR 识别
- **面板展示**：展开面板显示 OCR 识别的详细数据

**与语音播报模块联动：**

- **事件播报**：识别到击杀/死亡/助攻/控龙等事件时触发语音
- **段位提醒**：段位变化（升段/降段）时语音播报
- **异常告警**：OCR 识别失败/网络断开时语音提示
- **战术建议**：大模型分析结果通过语音输出

**与AI语音对话模块联动：**

- **屏幕理解**：用户问"现在什么情况"时，调用 OCR 识别当前画面并回答
- **上下文增强**：语音对话时自动附带最新 OCR 识别结果，提升回答准确性
- **可视化展示**：对话中展示 OCR 识别到的屏幕内容摘要

**启动流程（冷启动）：**

```
用户点击"开始辅助"
      ↓
[1] 权限检查
      ├─ 检查 MediaProjection 权限
      ├─ 检查悬浮窗权限
      └─ 检查通知权限
      ↓
[2] 权限申请（如未授权）
      ↓
[3] 初始化核心组件
      ├─ ScreenCaptureService 前台服务启动
      ├─ ImageReader 初始化
      ├─ MediaProjection 创建 VirtualDisplay
      ├─ WebSocket 连接（本地模式）
      ├─ 评分引擎初始化
      └─ 悬浮窗服务启动
      ↓
[4] 首帧预热
      ├─ 等待第一帧画面（超时 5s）
      ├─ 首帧 OCR 识别验证
      └─ 验证成功 → 进入正常运行
      ↓
[5] 正常运行
      ├─ 定时 OCR 识别
      ├─ 事件驱动识别
      ├─ 实时评分更新
      └─ UI/语音同步
```

**停止流程（正常停止）：**

```
用户点击"停止辅助"
      ↓
[1] 停止采集
      ├─ 停止帧采集循环
      ├─ 释放 ImageReader
      └─ 释放 VirtualDisplay
      ↓
[2] 停止推理
      ├─ 取消所有 pending 的 OCR 请求
      └─ 关闭 WebSocket 连接（本地模式）
      ↓
[3] 释放资源
      ├─ 回收所有 Bitmap 缓存
      ├─ 清空评分引擎状态
      ├─ 关闭悬浮窗（如配置）
      └─ 停止前台服务
      ↓
[4] 数据持久化
      ├─ 保存本局记录（如有对局）
      └─ 更新统计数据
      ↓
[5] 完全停止
      └─ 服务 onDestroy
```

**异常停止（崩溃/被系统杀死）：**

- 服务设置 `START_STICKY`，被杀死后自动重启
- 重启后从 SharedPreferences 恢复对话状态与配置
- 悬浮窗服务启动时检查并恢复未完成的对话
- 自动重连 WebSocket，恢复 OCR 识别

**资源释放规范：**

| 资源类型 | 释放时机 | 释放方式 | 责任人 |
|---------|---------|---------|--------|
| **Bitmap** | 每帧处理完成后 | `bitmap.recycle()` | 帧采集循环 |
| **Image** | 转换为 Bitmap 后立即 | `image.close()` | ImageReader |
| **MediaProjection** | 停止录屏时 | `mediaProjection.stop()` | Service |
| **VirtualDisplay** | 停止录屏时 | `virtualDisplay.release()` | Service |
| **ImageReader** | 停止录屏时 | `imageReader.close()` | Service |
| **WebSocket** | 停止服务/断连时 | 主动 close | 网络模块 |
| **协程作用域** | Service 销毁时 | `scope.cancel()` | Service |
| **悬浮窗** | 停止服务/配置关闭时 | removeView | 悬浮窗服务 |
| **前台通知** | 服务停止时 | `stopForeground(true)` | Service |

**内存安全规范：**

- 禁止在循环中创建新对象，尽量复用
- 所有 Bitmap 必须显式 recycle，禁止依赖 GC
- OCR 请求队列最大长度为 1，超出丢弃旧请求
- 监听系统 `onLowMemory()` 回调，触发时主动降频
- 定期（每 5 分钟）检查内存使用，超过阈值触发降级

## 十二、API接口设计

## 十六、安全性与隐私保护设计

### 1. 数据安全

- **本地存储加密**: 对局记录使用AES-256加密存储
- **网络传输加密**: WebSocket连接支持WSS（TLS加密）
- **敏感信息隔离**: API Key等敏感信息仅保存在本地，不上传云端

### 2. 隐私保护

- **无用户画像**: 不收集用户个人信息，设备ID为随机生成
- **数据可删除**: 提供一键清除所有本地数据的功能
- **开源透明**: 所有代码开源，接受社区审计

### 3. 合规保障

- **纯视觉方案**: 所有数据来自屏幕画面，不涉及游戏内存
- **无操作模拟**: 不模拟点击、不自动操作
- **仅信息展示**: AI分析结果仅用于展示和建议，不影响游戏运行
