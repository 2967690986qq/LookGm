# AI辅助-屏幕实时共享+DeepSeek-OCR结构化画面分析 产品功能说明

> 模块归属：全手游通用AI视觉助手 · 核心AI分析模块
>
> 版本：v1.0
>
> 适用游戏：王者荣耀（首发），可扩展至全系手游

## 一、功能定义与业务价值

### 1.1 功能定义

**AI辅助-屏幕实时共享+DeepSeek-OCR结构化画面分析**是全手游通用AI视觉助手的核心深度分析模块。基于Android系统级MediaProjection屏幕采集能力，结合手机端轻量CV前置过滤与PC端DeepSeek-OCR高精度结构化识别，实现游戏对局全流程关键指标的实时提取、智能评分与段位判定。

模块采用**"端侧轻量过滤 + 云端/本地大模型深度分析**的双层算力架构，手机端负责90%的基础状态识别（零网络消耗），DeepSeek-OCR仅处理10%的高精度结构化数值提取，在保证分析精度的同时将算力与带宽消耗降至最低。

### 1.2 业务价值

| 价值维度 | 具体描述 |
|---------|---------|
| **实时游戏辅助** | 对局中实时提取KDA、经济、输出/承伤占比、参团率等核心指标，玩家无需手动翻看面板即可掌握全局数据 |
| **智能段位评定** | 内置王者荣耀铜/银/金/顶级四段位判定引擎，实时计算本局总分，分数跨越档位触发弹窗+语音双重提醒 |
| **低算力成本** | 分层节流策略：本地轻量识别过滤静态画面，仅关键事件触发OCR推理，PC端模型日均Token消耗降低70%以上 |
| **双模式灵活部署** | 支持局域网本地部署（零费用、低延迟）与云端API调用（即开即用），用户按需切换 |
| **纯视觉合规** | 全程仅采集屏幕像素流，不触碰游戏内存、不修改游戏数据、不模拟操作，合规辅助工具边界清晰 |
| **模块化可扩展** | 结构化解析层与游戏解耦，新增手游仅需配置对应UI特征提取规则，无需重构核心引擎 |

## 二、完整业务运行流程

### 2.1 流程总览

```
用户开启AI辅助
    ↓
[第1步] 权限申请与校验
    ↓ （用户授权录屏/悬浮窗权限）
[第2步] 实时画面采集（MediaProjection）
    ↓ （连续YUV像素流 · 纯内存 · 无文件IO）
[第3步] 本地前置过滤（轻量CV识别）
    ├─ 画面差分检测 → 静止画面跳过
    ├─ 游戏状态识别 → 大厅/对线等低价值画面跳过
    └─ 关键事件检测 → 选人/团战/结算/语音提问时触发
    ↓ （仅高价值画面帧通过）
[第4步] 帧预处理（裁剪+压缩+编码）
    ├─ 裁剪：去除状态栏、悬浮球、桌面多余区域
    ├─ 缩放：对局画面自动缩放至720P
    └─ 编码：JPEG压缩 / H264硬件编码增量脏帧
    ↓ （二进制帧数据）
[第5步] WebSocket长连接传输
    ├─ 局域网模式 → PC端vLLM DeepSeek-OCR
    └─ 云端模式 → DeepSeek云端OCR API
    ↓ （结构化JSON结果）
[第6步] 数据解析与评分计算
    ├─ OCR文本结构化解析（KDA/经济/输出/承伤等）
    └─ 王者荣耀段位判定引擎实时评分
    ↓ （评分+评级+关键指标）
[第7步] 悬浮UI展示 + 语音播报联动
    ├─ 悬浮球实时显示评分/评级徽章
    ├─ 档位跨越触发弹窗提示
    └─ 关键事件语音播报
    └─ 对局数据持久化存储
```

### 2.2 各环节详细说明

#### 第1步：权限申请与校验

- **触发入口**：APP仪表盘页面「开启AI辅助」开关
- **权限类型**：
  - MediaProjection屏幕录屏权限（系统级弹窗授权）
  - 悬浮窗权限（国产手机需手动开启）
  - 前台服务权限（Android 8.0+）
- **权限拒绝处理**：弹窗引导用户手动开启，提供跳转设置页快捷入口
- **权限有效期**：一次授权，APP运行期间持续有效，应用被杀后重启需重新授权

#### 第2步：实时画面采集

- **技术方案**：Android MediaProjection + ImageReader + VirtualDisplay
- **采集方式**：连续YUV_420_888像素流，纯内存实时流转，不生成本地图片文件
- **采集分辨率**：与屏幕原生分辨率一致，后续预处理阶段统一缩放
- **采集帧率**：屏幕刷新率自适应（通常60fps），实际处理帧率由前置过滤层动态调节
- **首帧就绪**：启动后约100-300ms捕获第一帧，广播通知各模块截图已可用

#### 第3步：本地前置过滤（核心节流层）

**三层过滤机制，按优先级依次执行：**

| 过滤层级 | 检测方式 | 过滤逻辑 | 通过率 |
|---------|---------|---------|-------|
| **画面差分检测** | 像素级帧差法 | 对比前后两帧像素差异，差异阈值以下判定为静态画面，直接跳过 | ~40%画面被过滤 |
| **游戏状态识别** | 本地轻量CV模型 | 识别当前画面处于大厅/匹配/选人/对局/结算等状态，低价值状态不推送 | ~30%画面被过滤 |
| **关键事件触发** | 事件驱动机制 | 仅以下事件触发OCR推送：选人界面、团战爆发、击杀/死亡、结算面板、用户语音提问 | ~20%画面触发OCR |

**最终OCR触发频率**：约每3-10秒一次（依对局节奏动态调整）

#### 第4步：帧预处理

- **区域裁剪**：自动裁剪顶部状态栏、底部导航栏、自身悬浮球区域，仅保留游戏主画面
- **分辨率缩放**：对局画面统一缩放至1280×720（720P），平衡识别精度与传输体积
- **压缩编码**：JPEG质量系数动态调节（默认75%），单帧大小控制在50-150KB
- **脏帧优化**：仅传输画面变化区域（H.264硬件编码增量帧），静态场景带宽消耗降低90%

#### 第5步：OCR模型推理

**双部署模式**（详见第三章）：
- **局域网本地模式**：WebSocket二进制帧 → PC端vLLM部署的DeepSeek-OCR模型
- **云端API模式**：HTTP POST图片 → DeepSeek云端OCR API

**推理输出**：结构化JSON数据，包含游戏阶段、英雄、分路、KDA、经济、输出/承伤占比、控龙、推塔、治疗、视野、控制时长等全量可识别指标。

#### 第6步：数据解析与评分计算

- **OCR文本解析**：正则表达式 + 关键词匹配 + 数值归一化，将非结构化OCR文本转换为标准化对局数据结构
- **段位判定引擎**：基于王者荣耀铜/银/金/顶级四段位规则，实时计算本局总分（0-100分）
- **档位跨越检测**：检测分数是否跨越铜/银/金/顶级档位阈值，跨越时触发联动提醒
- **数据平滑**：连续3帧滑动平均，避免单帧识别误差导致评分跳变

#### 第7步：悬浮UI展示与联动

- **悬浮球状态显示**：实时OCR状态指示灯、当前评分、评级徽章
- **档位跨越提醒**：分数跨越档位阈值时，悬浮窗弹窗+语音双重提醒
- **对局数据存储**：每局对局数据本地持久化，支持历史战绩回看
- **语音播报联动**：关键事件（晋级、掉段、高光操作）自动语音播报

## 三、双部署模式

### 3.1 模式对比

| 对比维度 | 局域网本地DeepSeek-OCR | 云端DeepSeek OCR API |
|---------|---------------------|---------------------|
| **部署方式** | PC端vLLM本地部署模型 | 直接调用DeepSeek云端接口 |
| **费用成本** | 零费用（仅消耗本地算力/电费） | 按Token计费（约¥0.002-0.01/次） |
| **延迟表现** | 低延迟（局域网<100ms） | 较高延迟（公网500-2000ms） |
| **隐私安全** | 数据不出内网，隐私性高 | 图片上传云端，依赖服务商隐私政策 |
| **使用门槛** | 需要一台带显卡的PC，配置vLLM | 即开即用，仅需API Key |
| **适用场景** | 长期高频使用、隐私敏感用户 | 临时使用、无PC环境、外出场景 |
| **通信协议** | WebSocket二进制长连接 | HTTP POST短请求 |
| **并发能力** | 取决于PC显卡性能 | 云端弹性扩缩 |

### 3.2 局域网本地部署模式

#### 3.2.1 架构说明

手机与PC处于同一局域网（同一路由器），PC端通过vLLM部署DeepSeek-OCR模型，同时运行WebSocket中转服务，接收手机端推送的画面帧，调用本地模型推理后返回结构化JSON结果。

#### 3.2.2 PC端环境要求

| 组件 | 最低配置 | 推荐配置 |
|-----|---------|---------|
| **显卡** | NVIDIA RTX 3060 12GB | NVIDIA RTX 4090 24GB |
| **显存** | ≥10GB | ≥16GB |
| **内存** | ≥16GB | ≥32GB |
| **系统** | Windows 10/11 / Linux | Windows 11 / Ubuntu 22.04 |
| **网络** | 千兆局域网（WiFi 6 / 有线） | 千兆有线局域网 |

#### 3.2.3 vLLM启动命令

```bash
# 启动 vLLM 服务（OpenAI兼容API）
python -m vllm.entrypoints.openai.api_server \
  --model deepseek-ai/DeepSeek-OCR \
  --host 0.0.0.0 \
  --port 8000 \
  --trust-remote-code \
  --max-model-len 4096 \
  --gpu-memory-utilization 0.9
```

**关键参数说明**：
- `--host 0.0.0.0`：开放局域网访问，手机端可通过PC内网IP连接
- `--max-model-len 4096`：OCR任务上下文长度，兼顾图文混合输入
- `--gpu-memory-utilization 0.9`：显存使用率90%，预留余量防止OOM

#### 3.2.4 WebSocket中转服务

PC端同时运行FastAPI + WebSocket中转服务，负责：
1. 接收手机端WebSocket二进制帧（JPEG图片）
2. 请求队列管理（最大5路并发，超出丢弃旧请求）
3. 调用vLLM DeepSeek-OCR模型推理
4. 结构化JSON结果返回手机端
5. 服务状态监控与日志记录

**服务启动命令**：
```bash
python ocr_websocket_server.py \
  --host 0.0.0.0 \
  --port 8765 \
  --model deepseek-ai/DeepSeek-OCR
```

### 3.3 云端DeepSeek OCR API模式

#### 3.3.1 架构说明

手机端直接通过HTTP POST上传压缩后的图片帧至DeepSeek云端OCR API，获取结构化识别结果。无需本地部署模型，即开即用。

#### 3.3.2 调用方式

```
POST https://api.deepseek.com/v1/chat/completions
Content-Type: application/json
Authorization: Bearer {API_KEY}
```

**请求体**：
```json
{
  "model": "deepseek-ocr",
  "messages": [
    {
      "role": "user",
      "content": [
        {
          "type": "image_url",
          "image_url": { "url": "data:image/jpeg;base64,{base64_image}" }
        },
        {
          "type": "text",
          "text": "请分析这张王者荣耀截图，提取所有对局数据并以JSON格式输出..."
        }
      ]
    }
  ],
  "stream": false
}
```

#### 3.3.3 配置入口

APP「模型设置」页面 → 添加DeepSeek供应商 → 开启「视觉模型」→ 选择云端OCR模式，填入API Key即可。

## 四、性能节流与容错机制

### 4.1 分层节流策略

#### 4.1.1 手机端节流（五层节流）

| 节流层级 | 实现方式 | 节流效果 |
|---------|---------|---------|
| **第1层：画面差分过滤** | 像素级帧差法，对比前后帧像素差异，静态画面直接跳过 | 静止场景减少40-60%帧处理量 |
| **第2层：游戏状态过滤** | 本地轻量CV识别游戏状态，大厅/匹配/结算等低价值时段不推送OCR | 非对局时段减少80%推理请求 |
| **第3层：关键事件驱动** | 仅选人/团战/击杀/结算/语音提问等关键事件才触发OCR | 对局中减少70%推理请求 |
| **第4层：动态帧率调节** | 对局平稳期10秒/次，团战期3秒/次，自动调节OCR间隔 | 平均请求频率降低50% |
| **第5层：画面压缩优化** | 720P缩放 + JPEG质量动态调节 + 增量脏帧传输 | 单帧体积减少60-80% |

**综合节流效果**：相比每秒60帧采集 → 平均每3-10秒1次OCR推理，算力消耗降低95%以上。

#### 4.1.2 PC端节流（三层节流）

| 节流层级 | 实现方式 | 节流效果 |
|---------|---------|---------|
| **请求队列** | 最大5路并发队列，满队丢弃最旧请求 | 防止模型过载拥堵 |
| **请求合并** | 短时间内多次请求合并处理最新一帧 | 避免重复推理 |
| **结果缓存** | 相似画面复用最近一次结果（1秒内） | 重复画面零推理消耗 |

### 4.2 异常容错机制

#### 4.2.1 权限相关容错

| 异常场景 | 容错策略 | 用户体验 |
|---------|---------|---------|
| **录屏权限拒绝** | 弹窗说明权限用途，提供跳转设置页入口，开关自动复位 | 明确指引用户手动开启 |
| **悬浮窗权限缺失** | 检测到无权限时弹窗引导，提供一键跳转设置 | 不影响核心功能，仅悬浮UI降级 |
| **国产手机后台杀进程** | 前台服务 + START_STICKY + 白名单引导 + 电池优化关闭引导 | 提升后台存活率 |

#### 4.2.2 网络通信容错

| 异常场景 | 容错策略 | 降级方案 |
|---------|---------|---------|
| **PC模型离线** | WebSocket心跳检测，断连后自动重试（指数退避：1s/2s/4s/8s/15s，最大重试5次） | 降级为仅手机本地轻量识别，关闭深度结构化分析，保留基础悬浮播报 |
| **WebSocket连接断开** | 自动重连机制，重连期间缓存最新帧，重连成功后立即补发 | 用户无感知，短暂丢失深度分析 |
| **请求超时（30s）** | 超时自动取消请求，不阻塞后续帧处理 | 跳过本次评分更新 |
| **队列满溢出** | 丢弃最旧请求，只保留最新帧 | 确保实时性优先 |

#### 4.2.3 模型推理容错

| 异常场景 | 容错策略 |
|---------|---------|
| **OCR识别结果为空** | 记录警告日志，跳过本次评分更新，不闪退 |
| **JSON解析失败** | 捕获异常，记录原始OCR文本用于debug，跳过本次更新 |
| **数据异常（数值越界）** | 数据校验，异常值过滤/平滑处理，防止评分跳变 |
| **连续失败降级** | 连续5次失败后自动进入降级模式，OCR间隔拉长至30秒，减少无效请求 |
| **模型返回格式异常** | 容错解析，尽可能提取可用字段，缺失字段用最近有效值填充 |

### 4.3 性能优化专项

#### 4.3.1 手机发热控制

- **CPU降频策略：OCR推理期间CPU使用率控制在60%以下
- **帧率动态调节：电池温度>45℃时自动拉长OCR间隔
- **内存管理：帧缓存上限3帧，及时回收Bitmap资源
- **硬编码加速：H.264硬件编码，降低CPU占用

#### 4.3.2 PC模型拥堵控制

- **请求排队：最大并发5路，超出排队
- **批量推理：支持多帧批量处理，提升GPU利用率
- **显存监控：显存使用率>90%时拒绝新请求

## 五、与现有模块的联动关系

### 5.1 与王者荣耀段位判定引擎的联动

#### 5.1.1 数据流向

```
OCR结构化结果 → 对局数据模型(MatchData) → 评分引擎(ScoringEngine) → 评分结果(ScoreResult)
```

#### 5.1.2 联动机制

| 联动点 | 说明 |
|-------|------|
| **数据输入** | OCR提取的KDA、经济、输出/承伤占比、参团率、控龙、推塔等指标，实时喂入评分引擎 |
| **评分计算** | 评分引擎基于王者荣耀五位置、四段位规则，实时计算本局总分（0-100分） |
| **档位检测** | 每次评分更新后检测是否跨越铜/银/金/顶级档位阈值，跨越时触发提醒 |
| **对局重置** | OCR检测到新一局选人界面时，自动重置评分引擎状态，开启新对局 |
| **结算校准** | 结算面板OCR识别后，由评分引擎进行最终总分校准，生成对局报告 |

#### 5.1.3 OCR提取的评分指标映射

| OCR提取字段 | 评分引擎用途 | 权重占比 |
|------------|------------|---------|
| 击杀/死亡/助攻（KDA） | 基础参战能力评分 | 25% |
| 经济/经济转输出 | 发育能力评分 | 15% |
| 输出占比/输出伤害 | 输出贡献评分 | 20% |
| 承伤占比/承受伤害 | 生存/承伤评分 | 15% |
| 参团率 | 团队贡献评分 | 10% |
| 控龙数量 | 资源控制评分 | 5% |
| 推塔数 | 推进能力评分 | 5% |
| 治疗量/视野价值/控制时长 | 辅助专项评分 | 5% |

### 5.2 与悬浮球模块的联动

#### 5.2.1 悬浮球状态显示

| 状态 | 悬浮球表现 | 触发条件 |
|-----|-----------|---------|
| **OCR待机** | 灰色指示灯 | AI辅助开启，未检测到游戏 |

#### 5.2.2 实时数据展示

- **档位跨越弹窗**：分数跨越档位阈值时，悬浮窗弹出晋级/掉段提示

#### 5.2.3 联动接口

悬浮球通过LocalBroadcastManager接收OCR引擎广播：
- `ACTION_OCR_STATUS`：OCR状态变更（idle/running/paused/error）
- `ACTION_SCORE_UPDATE`：评分更新（分数、评级、KDA）

## 六、开关启停全生命周期资源管理规范

### 6.1 生命周期状态机

```
          用户开启AI辅助
              ↓
        [初始化阶段]
              ↓  权限校验 + 服务启动
        [运行阶段] ←───┐
              ↓      │ 暂停/恢复
        [停止阶段]     │
              ↓      │
        [资源释放] ───┘
              ↓
          完全停止
```

### 6.2 开启流程（启动规范）

#### 6.2.1 启动时序

| 步骤 | 操作内容 | 所属模块 | 超时 |
|-----|---------|---------|------|
| 1 | 权限校验（录屏+悬浮窗） | DashboardFragment | 5s |
| 2 | 启动ScreenCaptureService前台服务 | ScreenCaptureService | 2s |
| 3 | 初始化MediaProjection + VirtualDisplay | ScreenCaptureService | 1s |
| 4 | 启动ImageReader帧采集循环 | ScreenCaptureService | - |
| 5 | 初始化本地轻量CV识别器 | GameStateDetector | 1s |
| 6 | 启动帧差分检测 + 事件触发机制 | ScreenCaptureService | - |
| 7 | 连接WebSocket（局域网模式） | LocalOcrClient | 10s |
| 8 | 初始化OCR结构化分析引擎 | OcrStructuredAnalyzer | - |
| 9 | 启动ScreenOcrEngine定时调度器 | ScreenOcrEngine | - |
| 10 | 悬浮球切换至OCR运行状态 | FloatingWindowService | - |
| 11 | 语音播报："AI辅助已开启" | VoiceService | - |

**总启动时间**：正常情况下约3-5秒

#### 6.2.2 启动失败回滚

任何步骤失败时，按逆序释放已启动资源，确保无资源泄漏：
- 失败步骤N → 依次释放步骤N-1到步骤1的资源 → 开关复位 → 弹窗提示失败原因

### 6.3 运行阶段管理

#### 6.3.1 运行时资源上限

| 资源类型 | 上限规范 | 超限处理 |
|---------|--------|---------|
| **内存** | 帧缓存≤3帧Bitmap，总内存≤100MB | OOM前主动回收，拉长OCR间隔 |
| **CPU** | 持续使用率≤60% | 动态降低帧率，拉长OCR间隔 |
| **网络** | 上行带宽≤500KB/s | 降低JPEG质量，缩小分辨率 |
| **电池温度** | ≤45℃ | 超过后降低频率，极限情况提示降温 |

#### 6.3.2 健康检查机制

- **心跳检测**：WebSocket每10秒发送心跳，30秒无响应判定断连
- **帧率监控**：连续10秒无新帧，判定采集异常，尝试重启采集
- **错误计数**：连续失败5次进入降级模式，30秒后尝试恢复

### 6.4 停止流程（资源释放规范）

#### 6.4.1 停止时序

| 步骤 | 操作内容 | 所属模块 | 验证标准 |
|-----|---------|---------|--------|
| 1 | 停止ScreenOcrEngine定时调度 | ScreenOcrEngine | 无新的OCR请求发出 |
| 2 | 断开WebSocket连接 | LocalOcrClient | 连接状态为disconnected |
| 3 | 清空待处理请求队列 | LocalOcrClient | 队列长度=0 |
| 4 | 释放OCR分析引擎资源 | OcrStructuredAnalyzer | 无引用持有 |
| 5 | 停止帧差分检测 | ScreenCaptureService | 差分线程退出 |
| 6 | 停止本地轻量CV识别 | GameStateDetector | 识别线程退出 |
| 7 | 停止ImageReader采集 | ScreenCaptureService | 无新帧回调 |
| 8 | 释放VirtualDisplay | ScreenCaptureService | display=null |
| 9 | 停止MediaProjection | ScreenCaptureService | projection=null |
| 10 | 回收所有Bitmap帧缓存 | FrameCompressor | 缓存大小=0 |
| 11 | 停止前台服务，移除通知 | ScreenCaptureService | 服务销毁 |
| 12 | 悬浮球切换回普通状态 | FloatingWindowService | 状态同步 |
| 13 | 语音播报："AI辅助已关闭" | VoiceService | 播报完成 |

#### 6.4.2 资源释放检查清单

停止完成后必须满足以下全部条件，判定为无资源泄漏：

- [ ] MediaProjection已停止，无系统录屏图标
- [ ] VirtualDisplay已释放，无虚拟显示残留
- [ ] ImageReader已关闭，无帧回调
- [ ] WebSocket连接已断开，无网络连接
- [ ] 所有CoroutineScope已取消，无协程泄漏
- [ ] 所有Bitmap已回收，无内存泄漏
- [ ] 前台服务已停止，通知栏无录屏通知
- [ ] 悬浮球状态已复位
- [ ]  wake lock已释放
- [ ]  WiFi lock已释放
- [ ]  所有广播接收器已注销

### 6.5 异常场景下的资源管理

#### 6.5.1 APP被系统杀死

- **Service配置**：`START_STICKY`，系统自动重启服务
- **状态恢复**：服务重启后从SharedPreferences读取保存的会话状态
- **资源重建**：自动重建MediaProjection（需用户重新授权）
- **用户体验**：悬浮球显示"正在恢复"状态，恢复成功后自动继续

#### 6.5.2 网络断开重连

- 断开期间：保留所有本地资源，仅断开WebSocket
- 重连期间：缓存最新帧，重连成功后立即补发
- 重连失败：指数退避重试，最大5次，失败后进入降级模式

#### 6.5.3 页面切换/APP切后台

- 切后台：录屏服务继续运行（前台服务保活）
- 切回前台：UI状态自动同步，无需重新初始化
- 任务清除：对话进行中不停止服务，空对话自动停止

## 附录：王者荣耀评级引擎数据匹配伪代码

```python
def calculate_score(match_data, position):
    """
    计算王者荣耀对局评分
    match_data: OCR提取的对局数据
    position: 分路（对抗路/中路/发育路/游走/打野）
    返回: ScoreResult(总分, 评级, 各维度得分)
    """
    
    # 1. 基础KDA评分 (25%)
    kda_score = calculate_kda_score(
        match_data.kills,
        match_data.deaths,
        match_data.assists,
        position
    ) * 0.25
    
    # 2. 经济发育评分 (15%)
    economy_score = calculate_economy_score(
        match_data.gold,
        match_data.gold_per_min,
        match_data.team_gold_rank,
        position
    ) * 0.15
    
    # 3. 输出贡献评分 (20%)
    damage_score = calculate_damage_score(
        match_data.damage_dealt,
        match_data.damage_dealt_percent,
        position
    ) * 0.20
    
    # 4. 生存承伤评分 (15%)
    survival_score = calculate_survival_score(
        match_data.damage_taken,
        match_data.damage_taken_percent,
        match_data.deaths,
        position
    ) * 0.15
    
    # 5. 团队参团评分 (10%)
    participation_score = calculate_participation_score(
        match_data.participation_rate,
        match_data.assists,
        position
    ) * 0.10
    
    # 6. 资源控制评分 (5%)
    objective_score = calculate_objective_score(
        match_data.dragon_kills,
        match_data.tower_kills,
        position
    ) * 0.05
    
    # 7. 辅助专项评分 (5% - 仅游走位)
    support_score = 0
    if position == "游走":
        support_score = calculate_support_score(
            match_data.healing_done,
            match_data.vision_score,
            match_data.crowd_control_time
        ) * 0.05
    
    # 总分计算
    total_score = kda_score + economy_score + damage_score + \
                  survival_score + participation_score + \
                  objective_score + support_score
    
    # 评级判定
    if total_score >= 97:
        grade = "顶级"
    elif total_score >= 90:
        grade = "金牌"
    elif total_score >= 75:
        grade = "银牌"
    elif total_score >= 60:
        grade = "铜牌"
    else:
        grade = "无评级"
    
    return ScoreResult(total_score, grade, ...)
```
